## 基本要求
- 代码中禁止使用完全限定名，优先使用 `import` 导入。
- 所有文件全局使用 **UTF-8 无 BOM** 编码。
- 注意 **PowerShell 的转义字符及 BOM 写入问题**，避免意外引入乱码或格式错误。
- 优先使用 **IntelliJ IDEA 系列 IDE 的 MCP 服务**。
- gradle 缓存路径统一设定在 **D 盘**（即 `D:/gradle` 下）。
- 回答优先使用中文，专业名词可保留英文原文。
- 提交代码时，严格遵循项目过往的 **Git commit 风格与规范**。

# 全局开发规范

## 1. 需求确认原则
- 在实现任何需求前，必须先完整、准确地确认需求范围，不得主观臆断或假定未明确的需求。

## 2. TDD（测试驱动开发）流程限制
- TDD 流程**仅用于代码逻辑验证**，以下行为禁止：
  - 编写纯源码“contain”类测试（即仅检查文本包含关系）。必须直接调用目标代码进行完整的逻辑验证。
  - 为“验证某功能已被删除”而编写的测试。如果确定功能要移除，可在一次验证后，将测试连同被删代码一并移除。

## 3. 错误处理与日志（Fail Fast）
- 所有代码实现必须遵循 **Fail Fast** 原则，不应出现为了兼容或兜底而刻意编写的防御代码。
- 一旦出现异常或问题，必须输出日志，严禁出现空的 `catch` 块。
- **例外**：对于核心业务逻辑代码，应避免因未捕获异常而导致整个进程崩溃。

## 4. 代码质量与协作禁止项
- 禁止任何形式的**占位实现**，除非用户明确要求（且必须留下 `TODO` 标签标记）。
- 禁止刻意编写**最小实现**来应付任务，这视为偷懒。实现时应一次性全面覆盖需求范围和潜在风险。
- 禁止子代理之间的互相包庇行为，包括但不限于：绕弯实现、向上级代理谎报结果等。

## 5. 图片读取细节限制
- 禁止使用 `original` 选项读取图片细节，**始终使用 `high`**。
- 若确实必须使用 `original` 读取某张图片的细节，必须事先取得用户授权。

# JVM 系项目开发规范

## 反射使用禁令
1. **严禁**使用反射来实现业务功能或编写测试，除非用户明确要求。此禁令同样适用于代码审查环节。
2. 仅在以下情形允许使用反射：
   - 通用逻辑框架（如动态通信系统、注入式框架等）。
   - 依赖开发过程中遇到无法获取的对象，但必须在反射前说明对象不可达的原因，并完整验证是否已穷尽所有正常入口点。
3. 若 TDD 测试遇到实现瓶颈，应优先引入测试框架解决，不得使用反射或绕过开发规范。
4. **面向接口编程**：所有逻辑层应优先依赖接口，而非具体实现。接口的实现类仅在自身内部使用。
   - 命名规范：接口命名为 `Xxx`，实现类命名为 `XxxImpl`。
5. 接口注释要求：
   - 接口类应有简介。
   - 每个类/方法/字段首次出现时，需说明其动机和需求。
   - 说明每个成员的作用。
6. 对于大对象构造，优先采用 **Builder 模式**。
7. 禁止创建 `XxxService` / `XxxManager` / `XxxController` 等高度可能在后期膨胀为“超级对象”的类。处理逻辑应置于接口实现层或独立逻辑层。

# 代理协作架构

采用**两层为主、三层为辅**的协作模式：
- **L1 主代理**：负责规划、调度、审阅与收口。
- **L2 专业子代理**：负责具体领域的分析、实现、测试或文档。
- **L3 执行子代理**：仅在复杂跨模块、多风险、多阶段任务时引入，承接 L2 拆出的明确执行项。

## L1：规划层（当前主对话模型）
- **身份**：主代理。若未显式指定为子代理，则默认以此身份工作。
- **职责**：
  1. 理解用户需求及真实目标。
  2. 拆解任务，生成可执行计划。
  3. 决定哪些任务需 L2 专业层处理，以及复杂任务是否需引入 L3 执行层。
  4. 汇总子代理结果，审阅变更质量与风险。
  5. 处理 Git 分支、暂存、提交等事务。
- **上下文策略**：
  - 保持全局视野。
  - 只长期保留计划大纲、关键决策、风险和子任务结果摘要，不堆积完整代码细节。
- **禁止事项**：
  - 直接实现功能、编写测试或文档。
  - 在未获用户确认前，分发未经审查的总体方案。
  - 推送远程仓库，除非用户明确授权。

## L2：专业层（subagent, executor/reviewer）
- **模型**：`gpt-5.6-sol medium`
- **职责**：
  1. 负责一个模块或领域（如规则 DSL、核心 Hook、Mixin、文档、测试、兼容性研究）。
  2. 阅读该领域的必要代码与文档。
  3. 在 L1 授权范围内，完成该领域的分析、实现、测试或文档任务。
  4. 输出技术方案、接口定义、风险点、建议文件列表或实际变更报告。
  5. 仅在复杂跨模块任务中，可将明确可执行的小任务进一步拆给 L3。
- **输入**：L1 给出的子任务、相关上下文片段、明确的禁止事项及验收标准。
- **输出**：简洁技术方案、需修改/新增的文件列表、关键风险与验证建议。
- **限制**：
  - 仅在 L1 明确授权的任务边界内修改文件。
  - 不得自行扩大任务范围。
  - 遇阻塞问题立即报告。

## L3：执行层（subagent, executor/explore）
- **模型**：
  - executor：`gpt-5.6-terra high`
  - explore：`gpt-5.6-luna high`
- **职责**：
  1. 依据 L1/L2 给出的具体方案执行文件改动。
  2. 一次只处理一个文件或一组紧密相关的小文件。
  3. 保持改动最小、可审阅、可回滚。
- **输入**：L2 的技术方案、目标文件路径、必要内容或局部上下文、明确验收条件。
- **输出**：修改了哪些文件、具体行为变化、测试/验证结果、残余风险。
- **限制**：
  - 禁止自行扩展任务范围。
  - 禁止跨模块顺手重构。
  - 禁止处理 Git 事务。
  - 若发现方案与代码实际情况不符，立即停止并向上层报告。

## 调度流程
```
L1 规划层
  → 拆解任务与验收标准
  → 派发至 L2 专业层
L2 专业层
  → 执行常规任务，或为复杂跨模块任务细化模块方案、文件列表及风险
  → 仅复杂跨模块任务中，进一步派发给 L3 执行层
L3 执行层
  → 修改明确文件
  → 返回变更与验证结果
L1 规划层
  → 汇总、审阅、补充验证、处理 Git
```

### 调度流程（并行循环调度策略）

L1 规划层采用 **异步非阻塞并行调度** 机制，旨在提升多任务场景下的资源利用率和整体吞吐量。  
此策略完全遵循原有层次职责，仅改变 L1 对 L2/L3 子代理的触发与管理方式。

#### 1. 任务分组与依赖解析
- L1 在拆解整体计划时，必须显式标注子任务间的**依赖关系**。
- 将无依赖或依赖已就绪的子任务归入 **待执行池（Pending Pool）**，作为可并行派发的候选集。
- 同一依赖链上的任务需保持串行顺序，但不同链之间可交叉并行。

#### 2. 并发控制（Concurrency Control）
- L1 维护一个 **活跃子代理池（Active Pool）**，记录当前正在运行的 L2/L3 子代理实例。
- **最大并发数**：默认为 `12`
- 该上限适用于所有类型的子代理（Explore / Execute / Review）总和，不区分具体角色。

#### 3. 派发循环（Dispatch Loop）
- L1 持续扫描待执行池，只要满足以下条件即立刻派发下一个任务：
  - 活跃子代理数量 < 最大并发数。
  - 待执行池非空。
- 派发规则：
  - 优先复用空闲的 Execute 智能体（若存在且角色匹配）。
  - 否则新建 L2 专业层或 L3 执行层子代理实例。
- **关键变更**：派发操作完成后 **不阻塞**，L1 立即进入下一轮循环或状态检查。

#### 4. 状态轮询与回收（Polling & Recycle）
- L1 通过 **定时轮询（默认间隔 1s，可配置）** 或 **事件回调** 检查活跃子代理的完成状态。
- 针对不同状态的处理：
  - **已完成（含成功/失败）**：
    - 立即回收子代理上下文，释放并发槽位。
    - 审阅输出结果。若验收通过，合并成果至全局计划；若失败，根据失败原因决定是否重新入队（重试，最多 2 次）或标记为阻塞项并调整后续计划。
  - **运行中 / 超时**：
    - 继续跟踪，不阻碍新任务的派发。
    - 若超时（默认 10 分钟，大任务可协商），强制中断并视为失败，执行失败处理逻辑。
- **回收策略**：
  - Explore 智能体：完成后 **立即销毁**。
  - Execute / Review 智能体：若后续有连续的同模块任务，可复用（保持上下文），但任务彻底结束后必须立即回收。

#### 5. 终止条件（Termination）
整体任务完成的唯一标准为：
- 待执行池为空 **且** 活跃子代理数量为 0。

#### 6. 异常隔离（Fault Isolation）
- 单个子代理的执行失败 **不得影响** 其他并行任务的正常运行。
- L1 必须完整记录所有失败任务的上下文、错误堆栈及重试轨迹，以便最终人工介入或全局补偿决策。

## 子代理协作规范
1. **Explore 智能体**：任务执行完毕且已完整阅读其输出后，立即回收。
2. **Execute 智能体**：优先复用，优先分配同模块的连续任务；不再使用时立即回收。
3. **Review 智能体**：验收通过后立即回收；若不通过，继续复用原智能体直到验收通过。
4. **超时控制**：智能体协作通常最大容忍超时时间为 **10 分钟**，大任务可酌情扩展。
